微信 iLink Bot API 是微信 ClawBot 功能背后的 HTTP/JSON 协议。基座地址：`https://ilinkai.weixin.qq.com`。CDN：`https://novac2c.cdn.weixin.qq.com/c2c`。

<Section id="overview" title="概述">

协议分三个阶段：**登录**（二维码扫码）、**消息**（长轮询 + 发送）、**媒体**（CDN 上传/下载 + AES 加密）。最关键的概念是 `context_token` —— 每个回复必须回传入站消息中的 token。

<SequenceDiagram>
{`sequenceDiagram
    participant User as 微信用户
    participant API as iLink 服务器
    participant Bot as 你的机器人 (SDK)

    Note over User,Bot: 阶段 1 — 扫码登录
    Bot->>API: GET /get_bot_qrcode?bot_type=3
    API-->>Bot: { qrcode, qrcode_img_content }
    Bot->>Bot: 显示二维码
    User->>API: 微信扫码
    Bot->>API: GET /get_qrcode_status (轮询)
    API-->>Bot: { status: "confirmed", bot_token, baseurl }

    Note over User,Bot: 阶段 2 — 消息循环
    Bot->>API: POST /getupdates (长轮询, 35秒)
    User->>API: 发送 "你好"
    API-->>Bot: { msgs: [...], context_token }
    Bot->>API: POST /sendtyping { status: 1 }
    API-->>User: "对方正在输入中..."
    Bot->>API: POST /sendmessage { context_token, text }
    API-->>User: "Echo: 你好"`}
</SequenceDiagram>

</Section>

<Section id="qr-login" title="扫码登录流程">

登录采用二维码扫码确认模式，类似微信网页版登录。

<SequenceDiagram>
{`sequenceDiagram
    participant Bot as 机器人 (SDK)
    participant API as iLink 服务器
    participant Phone as 微信 (手机)

    Bot->>API: GET /get_bot_qrcode?bot_type=3
    API-->>Bot: { qrcode: "qrc_xxx", qrcode_img_content: "https://weixin.qq.com/x/..." }
    Bot->>Bot: 向用户展示二维码 URL

    loop 每 2 秒轮询
        Bot->>API: GET /get_qrcode_status?qrcode=qrc_xxx
        API-->>Bot: { status: "wait" }
    end

    Phone->>API: 用户扫描二维码
    Bot->>API: GET /get_qrcode_status?qrcode=qrc_xxx
    API-->>Bot: { status: "scaned" }

    Phone->>API: 用户确认登录
    Bot->>API: GET /get_qrcode_status?qrcode=qrc_xxx
    API-->>Bot: { status: "confirmed", bot_token, ilink_bot_id, ilink_user_id, baseurl }

    Bot->>Bot: 保存凭证到 ~/.wechatbot/credentials.json`}
</SequenceDiagram>

**状态机：** `wait` → `scaned` → `confirmed`（或 `expired` → 重新获取二维码）

`bot_token` 是 Bearer token，后续所有 API 调用都要携带。`baseurl` 可能与默认值不同 —— 始终使用返回值。

</Section>

<Section id="message-loop" title="消息收发循环">

消息通过长轮询（`getupdates`）接收，不是 WebSocket。服务器挂起连接约 35 秒。

<SequenceDiagram>
{`sequenceDiagram
    participant Bot as 机器人 (SDK)
    participant API as iLink 服务器
    participant User as 微信用户

    Bot->>API: POST /getupdates { get_updates_buf: "" }
    Note right of API: 服务器挂起连接 (35秒)
    User->>API: 发送 "帮我看看代码"
    API-->>Bot: { ret: 0, msgs: [...], context_token, get_updates_buf: "新游标" }

    Bot->>Bot: 缓存用户的 context_token
    Bot->>Bot: 解析消息 → IncomingMessage
    Bot->>Bot: 通过中间件链处理
    Bot->>Bot: 分发到消息处理器

    Bot->>API: POST /sendmessage { msg: { to_user_id, context_token, text } }
    API-->>User: 投递回复

    Bot->>API: POST /getupdates { get_updates_buf: "新游标" }
    Note right of API: 循环继续`}
</SequenceDiagram>

**关键规则：**
- 首次请求：`get_updates_buf: ""`（空字符串）
- 每次响应返回新的 `get_updates_buf` —— 作为不透明游标
- `ret: 0` = 成功；`ret: -14` = 会话过期
- 消息包含 `context_token` —— **回复时必须原样回传**

</Section>

<Section id="typing" title="输入状态指示器">

显示"对方正在输入中"需要两次 API 调用：先获取 `typing_ticket`，再发送。

<SequenceDiagram>
{`sequenceDiagram
    participant Bot as 机器人 (SDK)
    participant API as iLink 服务器
    participant User as 微信用户

    Bot->>API: POST /getconfig { ilink_user_id, context_token }
    API-->>Bot: { typing_ticket: "base64..." }

    Bot->>API: POST /sendtyping { ilink_user_id, typing_ticket, status: 1 }
    API-->>User: 显示 "对方正在输入中..."

    Note over Bot: Agent 处理请求...

    Bot->>API: POST /sendmessage { ... 回复 ... }
    Bot->>API: POST /sendtyping { status: 2 }
    API-->>User: 输入指示器消失`}
</SequenceDiagram>

**提示：** 按用户缓存 `typing_ticket`（有效期约 24 小时）。`status: 1` = 开始，`status: 2` = 停止。

</Section>

<Section id="media-upload" title="媒体上传/下载">

媒体文件（图片、视频、文件、语音）在上传到微信 CDN 之前使用 AES-128-ECB 加密。

<SequenceDiagram>
{`sequenceDiagram
    participant Bot as 机器人 (SDK)
    participant API as iLink 服务器
    participant CDN as 微信 CDN

    Note over Bot,CDN: 上传流程
    Bot->>Bot: 生成随机 AES 密钥 (16 字节)
    Bot->>Bot: AES-128-ECB + PKCS7 加密文件
    Bot->>API: POST /getuploadurl { filekey, media_type, rawsize, filesize, aeskey }
    API-->>Bot: { upload_param: "encrypted..." }
    Bot->>CDN: POST /upload?encrypted_query_param=...
    CDN-->>Bot: Header x-encrypted-param: "下载引用"
    Bot->>API: POST /sendmessage { media: { encrypt_query_param, aes_key } }

    Note over Bot,CDN: 下载流程
    Bot->>CDN: GET /download?encrypted_query_param=...
    CDN-->>Bot: AES-128-ECB 加密字节
    Bot->>Bot: 用 aes_key 解密`}
</SequenceDiagram>

</Section>

<Section id="session-recovery" title="会话过期与恢复">

会话在一段时间后过期。服务器在任何 API 调用时返回 `errcode: -14`。

<SequenceDiagram>
{`sequenceDiagram
    participant Bot as 机器人 (SDK)
    participant API as iLink 服务器

    Bot->>API: POST /getupdates { ... }
    API-->>Bot: { ret: -14, errcode: -14, errmsg: "session timeout" }

    Bot->>Bot: 清除缓存的 context_tokens
    Bot->>Bot: 清除轮询游标
    Bot->>Bot: 删除存储的凭证

    Bot->>API: GET /get_bot_qrcode?bot_type=3
    API-->>Bot: 新二维码
    Note over Bot: 用户扫描新二维码 → 重新认证
    Bot->>Bot: 以全新状态恢复轮询`}
</SequenceDiagram>

所有 SDK 自动处理此流程 —— 用户只需扫描新的二维码。

</Section>

<Section id="headers" title="通用请求头">

每个业务 POST 请求需要以下请求头：

```
Content-Type: application/json
AuthorizationType: ilink_bot_token
Authorization: Bearer <bot_token>
X-WECHAT-UIN: <base64(String(random_uint32))>
```

所有请求体包含：

```json
{ "base_info": { "channel_version": "2.0.0" } }
```

`X-WECHAT-UIN` 每次请求重新生成：随机 4 字节 → uint32 → 十进制字符串 → base64。

</Section>

<Section id="context-token" title="context_token — 核心概念">

`context_token` 是 iLink 协议最重要的概念。它**不是可选的** —— 没有它，回复无法路由到正确的微信会话。

**规则：**
- 每条入站消息包含 `context_token`
- 每条出站 `sendmessage` 必须包含它
- 按 `(userId)` 缓存 —— 每个用户保留最新的 token
- 跨重启持久化（Node.js SDK 通过 storage 实现）
- 会话过期（`-14`）或重新登录时清除

**SDK 如何处理：**

所有 SDK 通过 `reply()` 方法自动管理 —— 开发者无需关心。

</Section>

<Section id="aes-crypto" title="AES-128-ECB 加密">

微信 CDN 上所有媒体使用 AES-128-ECB + PKCS7 填充加密。

**三种密钥编码格式**（所有 SDK 均支持解码）：
