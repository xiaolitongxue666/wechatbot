The WeChat iLink Bot API is an HTTP/JSON protocol used by the WeChat ClawBot feature. Base URL: `https://ilinkai.weixin.qq.com`. CDN: `https://novac2c.cdn.weixin.qq.com/c2c`.

<Section id="overview" title="Overview">

The protocol has three phases: **login** (QR scan), **messaging** (long-poll + send), and **media** (CDN upload/download with AES encryption). The critical concept is `context_token` — every reply must echo it back from the incoming message.

<SequenceDiagram>
{`sequenceDiagram
    participant User as WeChat User
    participant API as iLink Server
    participant Bot as Your Bot (SDK)

    Note over User,Bot: Phase 1 — QR Login
    Bot->>API: GET /get_bot_qrcode?bot_type=3
    API-->>Bot: { qrcode, qrcode_img_content }
    Bot->>Bot: Display QR URL
    User->>API: Scan QR in WeChat
    Bot->>API: GET /get_qrcode_status (poll)
    API-->>Bot: { status: "confirmed", bot_token, baseurl }

    Note over User,Bot: Phase 2 — Message Loop
    Bot->>API: POST /getupdates (long-poll, 35s)
    User->>API: Send "Hello"
    API-->>Bot: { msgs: [...], context_token }
    Bot->>API: POST /sendtyping { status: 1 }
    API-->>User: "typing..."
    Bot->>API: POST /sendmessage { context_token, text }
    API-->>User: "Echo: Hello"`}
</SequenceDiagram>

</Section>

<Section id="qr-login" title="QR login flow">

Login uses a QR code scan-and-confirm pattern, similar to WeChat Web login.

<SequenceDiagram>
{`sequenceDiagram
    participant Bot as Bot (SDK)
    participant API as iLink Server
    participant Phone as WeChat (Phone)

    Bot->>API: GET /get_bot_qrcode?bot_type=3
    API-->>Bot: { qrcode: "qrc_xxx", qrcode_img_content: "https://weixin.qq.com/x/..." }
    Bot->>Bot: Show QR URL to user

    loop Poll every 2 seconds
        Bot->>API: GET /get_qrcode_status?qrcode=qrc_xxx
        API-->>Bot: { status: "wait" }
    end

    Phone->>API: User scans QR code
    Bot->>API: GET /get_qrcode_status?qrcode=qrc_xxx
    API-->>Bot: { status: "scaned" }

    Phone->>API: User confirms login
    Bot->>API: GET /get_qrcode_status?qrcode=qrc_xxx
    API-->>Bot: { status: "confirmed", bot_token, ilink_bot_id, ilink_user_id, baseurl }

    Bot->>Bot: Save credentials to ~/.wechatbot/credentials.json`}
</SequenceDiagram>

**Status machine:** `wait` → `scaned` → `confirmed` (or `expired` → request new QR)

The `bot_token` is a Bearer token used for all subsequent API calls. `baseurl` may differ from the default — always use the returned value.

</Section>

<Section id="message-loop" title="Message receive/send loop">

Messages are received via long-polling (`getupdates`), not WebSocket. The server holds the connection for ~35 seconds.

<SequenceDiagram>
{`sequenceDiagram
    participant Bot as Bot (SDK)
    participant API as iLink Server
    participant User as WeChat User

    Bot->>API: POST /getupdates { get_updates_buf: "" }
    Note right of API: Server holds connection (35s)
    User->>API: Sends "帮我看看代码"
    API-->>Bot: { ret: 0, msgs: [{ from_user_id, context_token, item_list }], get_updates_buf: "new_cursor" }

    Bot->>Bot: Cache context_token for user
    Bot->>Bot: Parse message → IncomingMessage
    Bot->>Bot: Run through middleware chain
    Bot->>Bot: Dispatch to message handlers

    Bot->>API: POST /sendmessage { msg: { to_user_id, context_token, item_list: [{ type: 1, text }] } }
    API-->>User: Delivers reply

    Bot->>API: POST /getupdates { get_updates_buf: "new_cursor" }
    Note right of API: Cycle repeats`}
</SequenceDiagram>

**Key rules:**
- First request: `get_updates_buf: ""` (empty string)
- Each response returns a new `get_updates_buf` — treat it as an opaque cursor
- `ret: 0` = success; `ret: -14` = session expired
- Messages contain `context_token` — **must echo it back in replies**

</Section>

<Section id="typing" title="Typing indicator flow">

Showing "对方正在输入中" requires two API calls: first get a `typing_ticket`, then send it.

<SequenceDiagram>
{`sequenceDiagram
    participant Bot as Bot (SDK)
    participant API as iLink Server
    participant User as WeChat User

    Bot->>API: POST /getconfig { ilink_user_id, context_token }
    API-->>Bot: { typing_ticket: "base64..." }

    Bot->>API: POST /sendtyping { ilink_user_id, typing_ticket, status: 1 }
    API-->>User: Shows "对方正在输入中..."

    Note over Bot: Agent processes request...

    Bot->>API: POST /sendmessage { ... reply ... }
    Bot->>API: POST /sendtyping { ilink_user_id, typing_ticket, status: 2 }
    API-->>User: Typing indicator disappears`}
</SequenceDiagram>

**Tips:** Cache `typing_ticket` per user (valid ~24h). `status: 1` = start, `status: 2` = stop. For long operations, send `status: 1` every 5 seconds as keepalive.

</Section>

<Section id="media-upload" title="Media upload/download">

Media files (images, videos, files, voice) are encrypted with AES-128-ECB before upload to the WeChat CDN.

<SequenceDiagram>
{`sequenceDiagram
    participant Bot as Bot (SDK)
    participant API as iLink Server
    participant CDN as WeChat CDN

    Note over Bot,CDN: Upload Flow
    Bot->>Bot: Generate random AES key (16 bytes)
    Bot->>Bot: Encrypt file with AES-128-ECB + PKCS7
    Bot->>API: POST /getuploadurl { filekey, media_type, rawsize, filesize, aeskey }
    API-->>Bot: { upload_param: "encrypted..." }
    Bot->>CDN: POST /upload?encrypted_query_param=...&filekey=...
    CDN-->>Bot: Header x-encrypted-param: "download_ref"
    Bot->>API: POST /sendmessage { item_list: [{ type: 2, image_item: { media: { encrypt_query_param, aes_key } } }] }

    Note over Bot,CDN: Download Flow
    Bot->>CDN: GET /download?encrypted_query_param=...
    CDN-->>Bot: AES-128-ECB encrypted bytes
    Bot->>Bot: Decrypt with aes_key from CDNMedia`}
</SequenceDiagram>

</Section>

<Section id="session-recovery" title="Session expiry & recovery">

Sessions expire after some time. The server returns `errcode: -14` on any API call when this happens.

<SequenceDiagram>
{`sequenceDiagram
    participant Bot as Bot (SDK)
    participant API as iLink Server

    Bot->>API: POST /getupdates { ... }
    API-->>Bot: { ret: -14, errcode: -14, errmsg: "session timeout" }

    Bot->>Bot: Clear cached context_tokens
    Bot->>Bot: Clear poll cursor
    Bot->>Bot: Delete stored credentials

    Bot->>API: GET /get_bot_qrcode?bot_type=3
    API-->>Bot: New QR code
    Note over Bot: User scans new QR → re-authenticated
    Bot->>Bot: Resume polling with fresh state`}
</SequenceDiagram>

All SDKs handle this automatically — the user just needs to scan a new QR code.

</Section>

<Section id="headers" title="Common request headers">

Every business POST request requires these headers:

```
Content-Type: application/json
AuthorizationType: ilink_bot_token
Authorization: Bearer <bot_token>
X-WECHAT-UIN: <base64(String(random_uint32))>
```

All request bodies include:

```json
{ "base_info": { "channel_version": "2.0.0" } }
```

`X-WECHAT-UIN` is generated fresh per request: random 4 bytes → uint32 → decimal string → base64.

</Section>

<Section id="context-token" title="context_token — the critical concept">

`context_token` is the most important concept in the iLink protocol. It's **not optional** — without it, replies cannot be routed to the correct WeChat conversation.

**Rules:**
- Every incoming message contains a `context_token`
- Every outgoing `sendmessage` must include it
- Cache per `(userId)` — the latest token for each user
- Persist across restarts (Node.js SDK does this via storage)
- Clear on session expiry (`-14`) or re-login
- Don't forge or reuse across users

**How SDKs handle it:**

```typescript
// Node.js — automatic via reply()
bot.onMessage(async (msg) => {
  // SDK extracts and caches msg._contextToken internally
  await bot.reply(msg, "Hello")  // SDK injects context_token automatically
})
```

```python
# Python — same pattern
@bot.on_message
async def handle(msg):
    await bot.reply(msg, "Hello")  # context_token managed by SDK
```

```go
// Go — same pattern
bot.OnMessage(func(msg *wechatbot.IncomingMessage) {
    bot.Reply(ctx, msg, "Hello")  // context_token managed by SDK
})
```

</Section>

<Section id="aes-crypto" title="AES-128-ECB encryption">

All media on the WeChat CDN is encrypted with AES-128-ECB + PKCS7 padding.

**Three key encoding formats** (all SDKs decode all three):

| Format | Example | Source |
|--------|---------|--------|
| base64(raw 16 bytes) | `ABEiM0RVZneImaq7zN3u/w==` | `CDNMedia.aes_key` (format A) |
| base64(hex string) | `MDAxMTIyMzM0NDU1NjY3Nzg4OTlhYWJiY2NkZGVlZmY=` | `CDNMedia.aes_key` (format B) |
| direct hex (32 chars) | `00112233445566778899aabbccddeeff` | `image_item.aeskey` |

**Decoding algorithm:**
1. If 32 hex chars → hex decode → 16 bytes
2. Else base64 decode → if 16 bytes, use directly
3. If 32 bytes and all hex ASCII → hex decode → 16 bytes

**Encrypted size:** `ceil((rawSize + 1) / 16) * 16`

</Section>

All SDKs follow the same design patterns established by **Telegraf** (middleware), **Discord.js** (EventEmitter), and **python-telegram-bot** (decorator handlers):

- **Event-driven** — register handlers, SDK dispatches
- **Auto context management** — `reply()` handles context_token, `send()` uses cached token
- **Session recovery** — automatic re-login on `-14`
- **Smart chunking** — text split at natural boundaries before send
- **Typing lifecycle** — start before processing, stop after reply

</Section>
